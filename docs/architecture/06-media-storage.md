# Media Storage Architecture

## Provider

Cloudflare R2 (S3-compatible, zero egress fees).

Supabase Storage was used in early development (Phase 4 MVP) but R2 is the target for all media at scale. The free plan gives 1 GB — insufficient for 100+ GB of photos and videos. R2 costs ~$1.50/month at 100 GB vs $25/month on Supabase Pro.

## Buckets

```txt
daemi-media (R2, private)
avatars (Supabase Storage, kept on free tier — small files only)
```

## Storage Path Convention

```txt
couple-spaces/{coupleSpaceId}/memories/{memoryId}/original.{ext}
couple-spaces/{coupleSpaceId}/memories/{memoryId}/thumb.{ext}
```

Same path convention applies to both R2 and legacy Supabase Storage objects.

## Auth Layer

R2 has no built-in RLS. Access control is handled by a Supabase Edge Function that:

1. Verifies the Supabase JWT.
2. Queries `couple_members` to confirm the user belongs to the couple space.
3. Returns a short-lived presigned URL (PUT for uploads, GET for downloads).

This replaces the Supabase Storage RLS policies with equivalent logic in the Edge Function.

## Upload Flow

1. User picks image with Expo Image Picker.
2. Client generates a UUID for the memory.
3. Client calls the `media-presign` Edge Function with `{ action: "upload", coupleSpaceId, memoryId, mimeType }`.
4. Edge Function verifies auth + membership, returns a presigned PUT URL (10 min expiry).
5. Client uploads the blob directly to R2 using the presigned URL.
6. Client inserts the memory row with `storage_key` (the R2 object path).
7. For display, client calls the Edge Function with `{ action: "download", coupleSpaceId, memoryId }` to get a presigned GET URL (1 hour expiry).

## Download Flow

Presigned GET URLs are generated on demand via the Edge Function. They are not stored in the database — only the `storage_key` (object path) is persisted. This eliminates the signed URL expiry problem.

## Database Schema

```sql
-- memories table includes:
storage_key text  -- R2 object path, e.g. "couple-spaces/{id}/memories/{id}/original.jpg"
media_url text    -- legacy: Supabase Storage signed URL (nullable, deprecated after migration)
```

## Security

- R2 bucket is private. No public access.
- All access goes through presigned URLs generated by the Edge Function.
- Edge Function verifies JWT + couple space membership before signing.
- Presigned URLs are short-lived: 10 min for uploads, 1 hour for downloads.
- R2 API credentials (access key, secret) are stored as Supabase Edge Function secrets — never exposed to the client.

## Migration from Supabase Storage

Phase 4 MVP uses Supabase Storage. Migration to R2:

1. Deploy `media-presign` Edge Function.
2. Update client upload code to use presigned URLs.
3. New uploads go to R2, old memories retain Supabase Storage URLs.
4. Backfill: script downloads from Supabase Storage, uploads to R2, updates `storage_key`.
5. Remove Supabase Storage bucket after backfill.

Both backends coexist during migration — the client checks `storage_key` first, falls back to `media_url`.

## Cost Estimate

| Item | 100 GB stored | 500 GB stored |
|---|---|---|
| R2 storage | $1.50/mo | $7.50/mo |
| R2 operations | ~$0.01/mo | ~$0.05/mo |
| R2 egress | $0 | $0 |
| Supabase free plan (auth + DB) | $0 | $0 |
| **Total** | **~$1.50/mo** | **~$7.55/mo** |
